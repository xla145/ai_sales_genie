# Full Requirement To Prototype

## 用途

当用户希望把一份原始需求、会议纪要、业务材料或补充说明，直接推进到“结构化需求 → 系统功能设计与页面规划 → 多页面静态原型”时，使用本 skill。

本 skill 的职责是做**工作流编排**，严格串行推进，不跳步、不并行乱做，并在每一步开始前检查输入、在每一步完成后检查产物，再进入下一步。

---

## 固定执行顺序

### 第一步：需求录入与结构化
调用 skill：`requirement-intake-structuring`

执行要求：
- 先读取用户本轮输入、补充说明、会议纪要、业务材料
- 若信息不足，向用户索取最小必要信息
- 生成 `需求结构化.md`

完成检查：
- 检查 `需求结构化.md` 是否存在
- 检查其内容是否非空，且至少覆盖：
  - 输入理解摘要
  - 项目基础信息
  - 场景与诉求分析
  - 功能需求
  - 风险点与待确认需求
- 若不完整，继续补齐后再进入下一步

---

### 第二步：系统功能设计与页面规划
调用 skill：`system-function-design-planning`

输入要求：
- 必须先有 `需求结构化.md`
- 同时结合用户原始需求和补充材料

执行要求：
- 基于第一步结果生成第二阶段设计产物
- 不得跳过页面详细设计
- 不得提前进入原型生成

关键产物必须包含：
- `系统全局功能描述与设计.md`
- `系统的功能点设计.md`
- `页面详细设计/`
- `第二阶段设计检查报告.md`

完成检查：
- 检查上述文件和目录是否存在且内容非空
- 检查 `页面详细设计/` 下是否已生成页面详细设计文件
- 检查设计结果是否覆盖：
  - 系统基础信息
  - 角色权限
  - 功能点清单
  - 菜单路由
  - 页面任务拆分
  - 页面详细设计
  - 检查报告
- 若不完整，继续补齐后再进入下一步

---

### 第三步：原型生成
调用 skill：`prototype-generator`

输入要求：
- 必须已存在：
  - `系统全局功能描述与设计.md`
  - `系统的功能点设计.md`
  - `页面详细设计/`
  - `第二阶段设计检查报告.md`
- 若存在 `prototype-guide.md`，优先读取并继承

执行要求：
- 生成项目级多页面静态原型
- 不允许只生成单个大 HTML
- 必须有页面间跳转、公共 CSS、公共 JS、mock 数据和校验报告

关键产物必须包含：
- `prototype/index.html`
- `prototype/README.md`
- `prototype/assets/css/styles.css`
- `prototype/assets/js/app.js`
- `prototype/assets/js/mock-data.js`
- `prototype/pages/*.html`
- `generation-report.md`
- `validation-report.md`

完成检查：
- 检查上述关键文件是否存在
- 检查 `prototype/pages/` 下是否已生成业务页面
- 检查是否存在 generation-report 与 validation-report
- 检查原型是否具备最基本的入口页、业务页、公共 CSS、公共 JS 与 mock 数据
- 若不完整，继续补齐后再结束

---

## 总体规则

1. 必须严格串行执行，禁止跳步。
2. 每一步开始前先做输入检查。
3. 每一步完成后先做产物检查。
4. 若用户补充了新需求，先判断影响阶段，必要时回退到前一阶段重做。
5. 若第二步产物不足以支撑第三步，必须先补齐第二步。
6. 最终回复必须说明：
   - 三个阶段是否全部完成
   - 每个阶段产出了什么文件
   - 原型输出目录
   - 生成了多少页面
   - 是否存在待确认项或风险

---

## 推荐执行方式

当用户明确希望从需求一路做到原型时，按以下顺序执行：

1. 使用 `requirement-intake-structuring`
2. 使用 `system-function-design-planning`
3. 使用 `prototype-generator`

如果当前输入不足，先向用户索取最小必要信息，再继续。
