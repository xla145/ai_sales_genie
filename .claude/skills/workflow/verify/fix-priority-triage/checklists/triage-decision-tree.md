# 问题分诊决策树

## 快速判定流程

```
发现问题
    │
    ├── 是否影响核心功能？
    │   ├── 是 → P0
    │   └── 否 → 继续
    │
    ├── 是否存在安全风险？
    │   ├── 是 → P0
    │   └── 否 → 继续
    │
    ├── 是否可能导致数据丢失？
    │   ├── 是 → P0
    │   └── 否 → 继续
    │
    ├── 是否导致编译/构建失败？
    │   ├── 是 → P0
    │   └── 否 → 继续
    │
    ├── 是否有明显性能退化？
    │   ├── 是 → P1
    │   └── 否 → 继续
    │
    ├── 是否有严重代码坏味道？
    │   ├── 是 → P1
    │   └── 否 → 继续
    │
    ├── 是否缺失关键异常处理？
    │   ├── 是 → P1
    │   └── 否 → 继续
    │
    ├── 核心逻辑是否缺少测试？
    │   ├── 是 → P1
    │   └── 否 → 继续
    │
    └── 其余问题 → P2
```

---

## P0 判定细则

| 问题类型 | 判定标准 | 示例 |
|----------|----------|------|
| 功能错误 | 核心业务流程无法完成 | 登录失败、支付无法完成 |
| 安全漏洞 | 可被利用的攻击面 | SQL 注入、XSS、CSRF |
| 数据风险 | 可能导致数据损坏或丢失 | 无事务的批量更新、无备份的 DDL |
| 编译失败 | 代码无法通过编译 | 语法错误、缺少依赖 |

## P1 判定细则

| 问题类型 | 判定标准 | 示例 |
|----------|----------|------|
| 性能问题 | 响应时间退化 >50% 或 N+1 查询 | 循环内数据库查询 |
| 严重坏味道 | 圈复杂度 >10 或方法 >80 行 | 上帝类、过长方法 |
| SOLID 违规 | 核心模块职责混乱 | Service 层包含 HTTP 逻辑 |
| 异常处理 | 可能导致静默失败 | 空 catch 块、吞掉异常 |
| 测试缺失 | 核心业务逻辑无测试覆盖 | 支付逻辑无单元测试 |

## P2 判定细则

| 问题类型 | 判定标准 | 示例 |
|----------|----------|------|
| 规范问题 | 不影响功能的代码风格 | 命名不规范、缺少 JavaDoc |
| 轻微坏味道 | 不影响可维护性的小问题 | 魔法数字、过长参数列表 |
| 可选优化 | 非必要的性能优化 | 缓存策略、批量查询 |
| 文档完善 | 文档缺失但不影响开发 | API 文档不完整 |

---

## P2 放行规则

| 条件 | 判定 | 处理 |
|------|------|------|
| P2 数量 ≤ 5 | ✅ 可放行 | 录入技术债务登记册 |
| P2 数量 6-10 | ⚠️ 建议修复部分 | 修复最严重的 5+ 个，其余放行 |
| P2 数量 > 10 | 🚫 不建议放行 | 需要整体重构或安排专项迭代 |

---

## 修复后触发检查

修复 P0/P1 问题后，必须执行以下检查：

1. **更新 design.md** — 如涉及接口或数据结构变更
2. **补充单元测试** — 修复逻辑必须有测试覆盖
3. **重新 Verify** — 确认修复有效且未引入新问题
4. **重新 Code Review** — 确认修复代码质量
