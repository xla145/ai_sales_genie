# 修复优先级分诊

## 规范来源

> 本 Skill 基于 Step4 修复优先级实现

## 触发条件

- vibe-verify 验收发现问题
- code-review-expert 审查发现问题
- CI/CD 构建或测试失败
- 用户提交问题清单需要优先级排序
- 迭代复盘时整理技术债务

## 工作流

### Phase 1: 问题收集与标准化

将来源不同的问题统一标准化格式：

```
问题来源
    │
    ├── vibe-verify 验收报告
    ├── code-review-expert 审查报告
    ├── CI/CD 失败日志
    ├── 手动测试反馈
    └── 用户反馈

标准化为统一格式：
    ├── # 编号
    ├── 问题描述
    ├── 发现来源
    ├── 影响范围
    ├── 当前状态
    └── 上下文信息（文件、行号、代码片段）
```

### Phase 2: P0/P1/P2 三级分诊

对每个问题进行优先级判定：

```
问题分诊
    │
    ├── 🚫 P0 — 阻断 MR，必须立即修复
    │   ├── 功能错误：核心功能无法使用
    │   ├── 安全漏洞：SQL注入、XSS、敏感信息泄露
    │   ├── 数据丢失风险：可能导致数据损坏或丢失
    │   └── 编译失败：代码无法编译或构建
    │
    ├── ⚠️ P1 — 阻断 MR，当次迭代修复
    │   ├── 性能问题：明显的性能退化
    │   ├── 严重坏味道：过长方法、上帝类、重复代码
    │   ├── 严重 SOLID 违规：核心模块的职责混乱
    │   ├── 缺失异常处理：可能导致静默失败
    │   └── 测试覆盖不足：核心逻辑无测试
    │
    └── 📝 P2 — 有条件放行，录入技术债务
        ├── 规范问题：命名不规范、注释缺失
        ├── 轻微坏味道：过长参数列表、魔法数字
        ├── 轻微 SOLID 违规：非核心模块的耦合
        ├── 可选优化：缓存策略、日志优化
        └── 文档完善：API 文档、使用说明
```

| 优先级 | 判定标准 | MR 处理 | 修复时限 |
|--------|----------|---------|----------|
| **P0** | 功能/安全/数据风险 | 🚫 阻断合并 | 立即修复 |
| **P1** | 性能/严重坏味道/可靠性 | 🚫 阻断合并 | 当次迭代内 |
| **P2** | 规范/轻微坏味道/优化 | ✅ 有条件放行（≤5 处） | 录入技术债务 |

### Phase 3: P2 有条件放行判定

对 P2 问题判断是否可以放行：

```
P2 放行条件判定
    │
    ├── P2 问题数量 ≤ 5 → ✅ 可放行
    │   └── 录入技术债务登记册
    │
    └── P2 问题数量 > 5 → ⚠️ 不建议放行
        └── 建议修复部分 P2，使剩余 ≤ 5
```

### Phase 4: 修复计划生成

为 P0 和 P1 问题生成修复计划：

```markdown
## 修复计划

### 🚫 P0 — 立即修复（阻断合并）

| # | 问题 | 位置 | 修复方案 | 预估工时 |
|---|------|------|----------|----------|
| 1 | [问题描述] | file:line | [修复方案] | [工时] |

### ⚠️ P1 — 当次迭代修复（阻断合并）

| # | 问题 | 位置 | 修复方案 | 预估工时 |
|---|------|------|----------|----------|
| 1 | [问题描述] | file:line | [修复方案] | [工时] |

### 📝 P2 — 技术债务（有条件放行）

| # | 问题 | 位置 | 录入债务 | 建议迭代 |
|---|------|------|----------|----------|
| 1 | [问题描述] | file:line | ✅ | 下次迭代 |
```

### Phase 5: 修复后触发链

修复完成后，自动触发后续检查：

```
修复完成
    │
    ├── 更新 design.md（如涉及接口或数据结构变更）
    │
    ├── 补充单元测试（修复逻辑需有测试覆盖）
    │
    └── 重新 Verify（确认修复有效，未引入新问题）
        ├── vibe-verify → 即时验收
        └── code-review-expert → 代码审查
```

### Phase 6: 技术债务登记

将 P2 问题录入技术债务登记册：

```markdown
## 技术债务登记

| # | 登记日期 | 问题描述 | 来源 | 优先级 | 建议迭代 | 状态 |
|---|----------|----------|------|--------|----------|------|
| 1 | YYYY-MM-DD | [描述] | code-review | P2 | Sprint N | 待处理 |
| 2 | YYYY-MM-DD | [描述] | vibe-verify | P2 | Sprint N+1 | 待处理 |
```

## 输入

| 输入项 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| 问题列表 | 列表 | 是 | 来自验收/审查/CI 的问题清单 |
| 问题来源 | 枚举 | 否 | vibe_verify / code_review / ci_cd / manual_test / user_feedback |
| 当前迭代 | 文本 | 否 | 当前 Sprint 编号，用于技术债务排期 |

## 输出

| 输出项 | 格式 | 说明 |
|--------|------|------|
| 分诊结果 | 表格 | P0/P1/P2 分类结果 |
| 修复计划 | 表格 | P0/P1 的修复方案和工时预估 |
| 放行判定 | ✅/⚠️ | P2 是否可放行 |
| 技术债务登记 | 表格 | P2 问题的债务记录 |
| 后续触发 | 列表 | 修复后需要执行的检查 |

## 检查清单

- [ ] 所有问题是否已按 P0/P1/P2 分类？
- [ ] P0 问题是否已制定立即修复方案？
- [ ] P1 问题是否已纳入当次迭代修复计划？
- [ ] P2 问题数量是否 ≤ 5？是否可放行？
- [ ] P2 问题是否已录入技术债务登记册？
- [ ] 修复后是否需要更新 design.md？
- [ ] 修复后是否需要补充单元测试？
- [ ] 修复后是否需要重新 Verify？
