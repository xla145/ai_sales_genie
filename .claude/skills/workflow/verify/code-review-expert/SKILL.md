# 代码审查专家

## 规范来源

> 本 Skill 基于 代码审查规范实现

## 触发条件

- MR 创建前/合并前代码审查
- 功能分支合并到 develop/release
- 核心模块变更
- 用户主动请求代码审查
- CI/CD 流水线触发

## 审查时机

| 场景 | 审查要求 | 审查者 |
|------|----------|--------|
| feature 合并到 develop | 必须审查 | 至少 1 人 |
| release 到 main | 必须审查 | 架构师 + 技术负责人 |
| hotfix 到 main | 必须审查 | 至少 1 人 |
| 核心模块变更 | 必须审查 | 技术负责人 |

## 工作流

### Phase 1: Preflight 上下文

- 使用 `git status -sb`、`git diff --stat`、`git diff` 获取变更范围
- 识别入口点、所有权边界和关键路径（认证、支付、数据写入、网络）
- 加载项目规范：`java-code-standards.md`、`security.md`

**边界情况：**
- **无变更**：提示用户选择审查暂存变更或指定 commit 范围
- **大 diff（>500 行）**：按文件/模块分组，分批审查
- **混合关注点**：按逻辑功能分组，而非仅按文件顺序

### Phase 2:  完整审查清单

对变更代码执行 7 维度审查：

#### 2.1 功能性

| 检查项 | 说明 | 严重级别 |
|--------|------|----------|
| 功能实现 | 是否实现需求？ | P0 |
| 逻辑正确性 | 逻辑是否正确？ | P0 |
| 边界条件 | 边界条件是否处理？ | P1 |
| 异常覆盖 | 异常情况是否覆盖？ | P1 |

#### 2.2 可读性

| 检查项 | 说明 | 严重级别 |
|--------|------|----------|
| 命名清晰 | 命名是否清晰有意义？ | P2 |
| 注释充分 | 关键逻辑是否有注释？ | P2 |
| 代码结构 | 代码结构是否清晰？ | P2 |

#### 2.3 质量性

| 检查项 | 说明 | 严重级别 |
|--------|------|----------|
| 重复代码 | 是否有重复代码？ | P1 |
| 过长方法 | 方法是否超过 50 行？ | P1 |
| 圈复杂度 | 圈复杂度是否 >10？ | P1 |

#### 2.4 性能性

| 检查项 | 说明 | 严重级别 |
|--------|------|----------|
| 性能问题 | 是否有明显的性能问题？ | P1 |
| 数据库查询 | 查询是否优化？N+1？ | P1 |
| 资源使用 | 资源使用是否合理？ | P2 |

#### 2.5 安全性

| 检查项 | 说明 | 严重级别 |
|--------|------|----------|
| 注入风险 | SQL 注入、XSS、命令注入？ | P0 |
| 输入验证 | 输入是否验证？ | P0 |
| 敏感数据 | 敏感数据是否保护？ | P0 |
| 认证授权 | 权限校验是否完整？ | P1 |

#### 2.6 测试性

| 检查项 | 说明 | 严重级别 |
|--------|------|----------|
| 单元测试 | 是否有单元测试？ | P1 |
| 覆盖率 | 覆盖率是否达标？ | P1 |
| 边界测试 | 边界条件是否覆盖？ | P2 |

#### 2.7 AI 生成代码专项（ 新增）

| 检查项 | 说明 | 严重级别 |
|--------|------|----------|
| AI 代码验证 | AI 生成代码是否经过验证？ | P0 |
| 人工审核签字 | 是否有人工审核确认？ | P1 |
| 规范符合 | 是否符合团队编码规范？ | P1 |
| 幻觉检测 | AI 是否生成了不存在的 API/库？ | P0 |
| 逻辑合理性 | AI 生成的逻辑是否合理？ | P1 |

### Phase 3: SOLID + 架构分析

加载 `references/solid-checklist.md`，深度检查：

- **SRP**：职责过载的模块
- **OCP**：频繁修改而非扩展
- **LSP**：子类破坏约定
- **ISP**：宽接口含未使用方法
- **DIP**：高层逻辑绑定底层实现

重构建议需说明 *为什么* 改善内聚/耦合，并提供增量计划。

### Phase 4: 安全与可靠性扫描

加载 `references/security-checklist.md`，检查：

- XSS、注入（SQL/NoSQL/命令）、SSRF、路径遍历
- AuthZ/AuthN 缺口、缺失租户检查
- 密钥泄露或 API Key 在日志/环境变量/文件中
- 限流、无限循环、CPU/内存热点
- 不安全反序列化、弱加密、不安全默认值
- **竞态条件**：并发访问、check-then-act、TOCTOU、缺失锁

### Phase 5: 代码质量扫描

加载 `references/code-quality-checklist.md`，检查：

- **错误处理**：吞掉异常、过宽 catch、缺失错误处理、异步错误
- **性能**：N+1 查询、热路径 CPU 密集操作、缺失缓存、无限内存
- **边界条件**：null/undefined 处理、空集合、数值边界、差一错误

### Phase 6: MR 门禁判定

根据审查结果，判定 MR 是否可以通过门禁：

```
MR 门禁判定
    │
    ├── P0 问题数 > 0 → 🚫 阻断合并
    │   └── 必须修复所有 P0 问题
    │
    ├── P1 问题数 > 0 → 🚫 阻断合并
    │   └── 必须修复所有 P1 问题
    │
    ├── P2 问题数 > 5 → ⚠️ 不建议合并
    │   └── 建议修复部分 P2，使剩余 ≤ 5
    │
    └── P2 问题数 ≤ 5 → ✅ 可以合并
        └── P2 录入技术债务登记册
```

| 门禁结果 | 条件 | 后续动作 |
|----------|------|----------|
| 🚫 **阻断** | 存在 P0/P1 | 修复后重新审查 |
| ⚠️ **有条件通过** | P2 > 5 | 修复部分 P2 |
| ✅ **通过** | P0=0, P1=0, P2≤5 | P2 录入技术债务 |

### Phase 7: 输出审查报告

```markdown
## 代码审查报告

**审查文件**：X 个文件，Y 行变更
**审查结果**：[APPROVE ✅ / REQUEST_CHANGES 🚫 / CONDITIONAL_APPROVE ⚠️]
**MR 门禁**：[通过 / 阻断]

---

## 发现问题

### P0 - Critical（阻断合并）
(none or list)

### P1 - High（阻断合并）
- **[file:line]** 简要标题
  - 问题描述
  - 修复建议

### P2 - Medium（有条件放行）
...

### AI 生成代码专项
- **[file:line]** AI 代码验证
  - 检查结果
  - 人工审核要求

---

## MR 门禁判定

| 指标 | 数量 | 门禁标准 | 状态 |
|------|------|----------|------|
| P0 问题 | N | = 0 | ✅/🚫 |
| P1 问题 | N | = 0 | ✅/🚫 |
| P2 问题 | N | ≤ 5 | ✅/⚠️ |

---

## 后续步骤

1. 修复所有 P0/P1 问题
2. 重新执行代码审查
3. P2 问题录入技术债务登记册
```

### Phase 8: 确认后续步骤

```markdown
---

## 后续步骤

发现 X 个问题（P0: _, P1: _, P2: _）。

**如何处理？**

1. **全部修复** — 我来实现所有建议的修复
2. **仅修复 P0/P1** — 只处理阻断合并的问题
3. **修复指定项** — 告诉我需要修复哪些
4. **仅审查** — 不需要实现修改

**重要**：在用户明确确认前，不实现任何修改。
```

## 输入

| 输入项 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| 审查范围 | 枚举 | 否 | git_changes（默认）/ file / directory / commit_range |
| 审查重点 | 枚举 | 否 | full（默认）/ security / quality / ai_specific / solid |
| MR 门禁模式 | 布尔 | 否 | 是否执行 MR 门禁判定，默认 true |

## 输出

| 输出项 | 格式 | 说明 |
|--------|------|------|
| 审查结果 | APPROVE/REQUEST_CHANGES/CONDITIONAL_APPROVE | 审查判定 |
| 问题列表 | 分级表格 | P0/P1/P2 分类问题 |
| AI 专项检查 | 表格 | AI 生成代码专项审查结果 |
| MR 门禁判定 | 表格 | 门禁通过/阻断判定 |
| 修复建议 | 文本 | 各问题的修复方案 |

## 检查清单

- [ ] 功能是否实现需求？逻辑是否正确？
- [ ] 边界条件和异常情况是否覆盖？
- [ ] 命名是否清晰？注释是否充分？
- [ ] 是否有重复代码或过长方法？
- [ ] 圈复杂度是否 >10？
- [ ] 是否有注入风险或安全漏洞？
- [ ] 是否有单元测试？覆盖率是否达标？
- [ ] AI 生成代码是否经过验证和人工审核？
- [ ] P0/P1 问题是否清零？
- [ ] P2 问题是否 ≤ 5？
