# MR 门禁判定清单

## 门禁标准

| 门禁项 | 通过标准 | 阻断条件 |
|--------|----------|----------|
| P0 问题数 | = 0 | > 0 |
| P1 问题数 | = 0 | > 0 |
| P2 问题数 | ≤ 5 | > 5 |
| 单元测试 | 通过 | 失败 |
| 覆盖率 | 核心业务 ≥ 80%，整体 ≥ 70% | 不达标 |
| 代码审查 | 至少 1 人批准 | 无批准 |

## 判定流程

```
MR 提交
    │
    ├── CI 流水线检查
    │   ├── 单元测试是否通过？ → 否 → 🚫 阻断
    │   └── 覆盖率是否达标？ → 否 → 🚫 阻断
    │
    ├── 代码审查
    │   ├── P0 > 0？ → 是 → 🚫 阻断
    │   ├── P1 > 0？ → 是 → 🚫 阻断
    │   ├── P2 > 5？ → 是 → ⚠️ 有条件通过
    │   └── P2 ≤ 5？ → 是 → ✅ 通过
    │
    └── 人工审批
        ├── 至少 1 人批准？ → 否 → 🚫 阻断
        └── 核心模块需技术负责人批准
```

## 审查时机门禁

| 场景 | 审查要求 | 审查者 |
|------|----------|--------|
| feature → develop | 必须审查 | 至少 1 人 |
| release → main | 必须审查 | 架构师 + 技术负责人 |
| hotfix → main | 必须审查 | 至少 1 人 |
| 核心模块变更 | 必须审查 | 技术负责人 |

## P2 技术债务处理

| 条件 | 处理 |
|------|------|
| P2 ≤ 5 | ✅ 录入技术债务登记册，可合并 |
| P2 6-10 | ⚠️ 修复最严重的 5+ 个，其余录入 |
| P2 > 10 | 🚫 不建议合并，需整体重构 |
