# 开发任务拆分

## 规范来源

> 本 Skill 基于 Step1 Vibe Coding 开发清单拆分要求实现

## 核心原则

> 氛围编程强调"把大需求拆成可执行任务，避免一次生成一坨不可控代码"

| 原则 | 说明 |
|------|------|
| **粒度适当** | 每个任务最好能在 1 次对话或 1 个开发 session 内完成 |
| **MVP 优先** | 先做 MVP 功能，后置可选功能 |
| **即时验收** | 每完成一个功能，立即验收，不符合就重新做 |
| **标准化** | 每个任务用统一模板描述 |

## 触发条件

- PRD 已确认，需要拆分开发任务
- 用户要求"拆分任务"或"制定开发清单"
- 变更提案（openspec:propose）前需要任务清单
- 用户说"这个需求太大了，帮我拆一下"

## 工作流

### Phase 1: 需求分析

从 PRD 中提取可拆分的功能点：

```
PRD 功能范围
    │
    ├── ✅ 必做（MVP）→ 优先拆分
    ├── 🔵 可选       → 后续拆分
    └── 🚫 暂不做     → 不拆分
```

### Phase 2: 四维度拆分

按 4 个维度交叉拆分，确保任务覆盖完整：

| 维度 | 说明 | 示例 |
|------|------|------|
| **按阶段拆** | 把大需求按开发阶段拆分 | 阶段 1: 静态页面 → 阶段 2: 交互逻辑 → 阶段 3: 数据存储 |
| **按模块拆** | 按功能模块独立拆分 | 登录注册、首页、列表页、详情页、API 接口 |
| **按角色拆** | 按开发角色拆分任务 | 前端任务、后端任务、测试任务 |
| **按优先级拆** | MVP 优先，后置可选 | MVP 功能 → 可选功能 → 暂不做功能 |

**拆分策略**：
1. 先按模块拆（确定边界）
2. 再按阶段拆（确定顺序）
3. 然后按角色拆（确定分工）
4. 最后按优先级排序（确定节奏）

### Phase 3: 任务标准化

每个任务使用标准模板：

```markdown
任务：[任务名称]
目标：[具体要达成的目标]
输入：[需要的输入，如设计稿、API 文档、对标截图]
输出：[预期的产出，如可访问的页面、可调用的 API]
依赖：[前置依赖，如设计风格已确定、接口已定义]
完成标准：[可验收的标准，如 PC 和移动端可正常显示、按钮可点击]
```

### Phase 4: 任务排序

按依赖关系和优先级排序：

```
排序规则：
1. 无依赖的任务优先
2. MVP 功能优先于可选功能
3. 后端接口优先于前端对接
4. 基础功能优先于高级功能
```

输出分阶段任务清单：

```markdown
## 开发任务清单

### 阶段 1：基础框架（MVP 核心）

| 序号 | 任务 | 角色 | 预估 | 依赖 | 完成标准 |
|------|------|------|------|------|----------|
| 1.1 | [任务] | [前端/后端] | [1-3天] | 无 | [标准] |
| 1.2 | [任务] | [前端/后端] | [1-3天] | 1.1 | [标准] |

### 阶段 2：功能完善

| 序号 | 任务 | 角色 | 预估 | 依赖 | 完成标准 |
|------|------|------|------|------|----------|

### 阶段 3：可选功能

| 序号 | 任务 | 角色 | 预估 | 依赖 | 完成标准 |
|------|------|------|------|------|----------|
```

### Phase 5: 粒度检查

检查每个任务的粒度是否合适：

| 检查项 | 合格标准 | 不合格处理 |
|--------|----------|------------|
| 任务时长 | ≤ 3 人天 | 超过则继续拆分 |
| 任务目标 | 单一明确 | 模糊则重新定义 |
| 完成标准 | 可验收 | 不可验收则补充 |
| 依赖关系 | 明确标注 | 循环依赖则调整顺序 |

## 输入

| 输入项 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| PRD 文档 | 文本 | 是 | 10 要素 PRD |
| 模块划分方案 | 文本 | 否 | 架构师提供的模块拆分 |
| 团队配置 | 文本 | 否 | 前后端人员分配情况 |

## 输出

| 输出项 | 格式 | 说明 |
|--------|------|------|
| 开发任务清单 | Markdown | 分阶段、分角色的任务列表 |
| 依赖关系图 | 文本 | 任务间的前后依赖 |
| 风险提示 | 列表 | 可能影响交付的风险点 |

## 检查清单

- [ ] 每个任务粒度是否 ≤ 3 人天？
- [ ] 每个任务是否有明确的完成标准？
- [ ] MVP 功能是否排在最前面？
- [ ] 依赖关系是否标注？
- [ ] 是否有循环依赖？
- [ ] 前后端任务是否对齐（接口先行）？
