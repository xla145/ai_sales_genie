# 需求确认稿生成

## 规范来源

> 本 Skill 基于 需求确认稿必须确认的 6 件事实现

## 触发条件

- Vibe Coding 需求访谈完成后
- 用户要求"生成需求确认稿"或"整理需求"
- 用户提供了零散的需求信息，需要结构化整理
- PRD 生成前，需要先确认需求边界

## 工作流

### Phase 1: 信息收集

从以下来源收集需求信息：

| 来源 | 内容 | 优先级 |
|------|------|--------|
| 访谈记录 | AI 与用户的问答记录 | 最高 |
| 用户提供的原始描述 | 用户最初的需求说明 | 高 |
| 参考文档 | 竞品分析、业务文档 | 补充 |
| 项目上下文 | AGENTS.md 中的项目背景 | 补充 |

### Phase 2: 信息提取与分类

将收集的信息按 6 件事分类提取：

```
原始信息
    │
    ├── 产品目标 ──── 为什么要做？解决什么问题？核心目标？
    ├── 目标用户 ──── 谁用？什么类型？技术背景？
    ├── 核心场景 ──── 怎么进入？主要任务？完整路径？
    ├── MVP 范围 ──── 必须做什么？可选做什么？
    ├── 暂不包含 ──── 明确不做什么？为什么？
    └── 待确认问题 ── 还不确定的？影响方案的？
```

**信息不足处理**：
- 对缺失的关键信息，标注为 `[待补充]`
- 对模糊的信息，标注为 `[需澄清：具体问题]`
- 对矛盾的信息，标注为 `[需确认：矛盾描述]`

### Phase 3: 生成确认稿

按标准模板生成需求确认稿：

```markdown
# 需求确认稿：[需求名称]

> 确认日期：[日期]

## 1. 产品目标

| 项目 | 内容 |
|------|------|
| 要解决的问题 | [一句话描述痛点] |
| 为什么要做 | [业务价值/用户价值] |
| 第一版核心目标 | [可量化的目标] |

## 2. 目标用户

| 用户类型 | 描述 | 技术背景 |
|----------|------|----------|

## 3. 核心使用场景

### 场景 1：[场景名称]
用户进入 → [入口] → 主要任务 → [步骤] → 完成目标 → [结果]

## 4. MVP 范围

| 优先级 | 功能 | 说明 |
|--------|------|------|
| ✅ 必做 | [功能] | [描述] |
| 🔵 可选 | [功能] | [描述] |

## 5. 暂不包含

| 功能 | 原因 | 预计迭代 |
|------|------|----------|

## 6. 待确认问题

| # | 问题 | 影响范围 | 可选方案 | 建议 |
|---|------|----------|----------|------|
```

### Phase 4: 对齐确认

生成确认稿后，执行对齐检查：

1. **复述理解** — 用自己的话简要概括需求
2. **标注不确定项** — 列出仍需确认的问题
3. **请求确认** — 等待用户逐项确认或修正

**输出格式**：
```
以上是我对需求的理解。请逐项确认：

✅ 已确认：[产品目标、目标用户...]
⚠️ 需补充：[具体缺失项]
❓ 需决策：[待确认问题]

确认后可进入 PRD 生成（/prd-generator）或任务拆分（/task-decomposition）。
```

## 输入

| 输入项 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| 访谈记录/需求描述 | 文本 | 是 | 访谈问答记录或用户原始需求 |
| 项目上下文 | 文本 | 否 | AGENTS.md 内容 |
| 参考文档 | 文本 | 否 | 竞品分析、业务文档链接 |

## 输出

| 输出项 | 格式 | 说明 |
|--------|------|------|
| 需求确认稿 | Markdown | 6 件事结构化确认稿 |
| 缺失项 | 列表 | 未能从输入中提取的信息 |
| 待确认问题 | 列表 | 需要用户决策的问题 |

## 检查清单

确认稿完成前逐项检查：

- [ ] 产品目标是否清晰可量化？
- [ ] 目标用户是否明确（类型+背景）？
- [ ] 核心场景是否完整（进入→操作→完成）？
- [ ] MVP 范围是否界定（必做 vs 可选 vs 不做）？
- [ ] 暂不包含是否明确（防止范围蔓延）？
- [ ] 待确认问题是否全部标注？
- [ ] 是否有 `[待补充]` 或 `[需澄清]` 标记？
