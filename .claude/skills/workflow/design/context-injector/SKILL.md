# 上下文注入器

## 规范来源

> 本 Skill 基于 上下文工程实现

## 触发条件

- AI 开始一个新任务或新对话
- 用户要求"注入上下文"或"加载项目信息"
- 切换到新的模块或功能开发
- AI 发现上下文缺失时主动请求

## 工作流

### Phase 1: 上下文探测

扫描项目目录，探测可用的上下文文件：

```
上下文探测
    │
    ├── 项目根目录
    │   ├── AGENTS.md       → 团队级上下文（必须）
    │   └── PROJECT.md      → 项目级上下文（如有）
    │
    ├── docs/design/         → 模块级 design.md
    │   └── {模块名}/
    │       └── design.md
    │
    └── .claude/
        ├── rules/           → 全局规则（自动加载）
        └── skills/          → Skills 规范（按需加载）
```

### Phase 2: 上下文组装

按层级组装上下文，从宏观到微观：

| 优先级 | 层级 | 文件 | 加载策略 | 必要性 |
|--------|------|------|----------|--------|
| 1 | 团队级 | `AGENTS.md` | 全量加载 | 必须 |
| 2 | 项目级 | `PROJECT.md` | 全量加载 | 如有 |
| 3 | 模块级 | `design.md` | 加载当前任务相关模块 | 必须（如存在） |
| 4 | 任务级 | TB 任务描述 | 用户提供的任务信息 | 如有 |
| 5 | 规则级 | `.claude/rules/` | 自动加载 | 自动 |

### Phase 3: 上下文格式化

按标准模板格式化组装的上下文：

```markdown
## 项目上下文

### 1. 项目概述
[从 AGENTS.md 提取项目概述]

### 2. 当前迭代目标
- 迭代版本：[如 v2.1]
- 迭代周期：[如 2026-04-15 ~ 2026-06-06]
- 本迭代重点：[核心交付目标]

### 3. 技术栈
[从 AGENTS.md 提取技术栈]

### 4. 项目结构
[从 AGENTS.md 提取项目结构]

### 5. 核心模块
[从 AGENTS.md 提取模块列表，标注当前任务相关模块]

### 6. 架构约束
[从 AGENTS.md 提取架构约束]

### 7. 当前任务
- 任务：[任务名称]
- 目标：[任务目标]
- 约束：[技术约束/业务约束]
- 参考文档：[design.md 链接]

### 8. 编码规范要点
[从 rules/ 提取关键规范摘要]
```

### Phase 4: 上下文验证

组装完成后，执行验证检查清单：

- [ ] 是否引用了正确的模块/类名？
- [ ] 是否符合项目的编码规范？
- [ ] 是否与 design.md 中的设计一致？
- [ ] 是否考虑了依赖关系？
- [ ] 是否有上下文缺失需要补充？

**验证失败处理**：
- 标注缺失项
- 向用户请求补充信息
- 重新组装直到验证通过

### Phase 5: 上下文注入

将格式化后的上下文注入到当前对话中：

1. 输出结构化的项目上下文
2. 确认用户理解当前上下文
3. 开始执行任务

## 输入

| 输入项 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| 任务描述 | 文本 | 是 | 当前要执行的任务 |
| 模块名称 | 文本 | 否 | 任务所属模块，用于定位 design.md |
| 额外上下文 | 文本 | 否 | 用户提供的补充信息 |

## 输出

| 输出项 | 格式 | 说明 |
|--------|------|------|
| 项目上下文 | Markdown | 结构化的项目上下文 |
| 缺失项 | 列表 | 未能获取的上下文项 |
| 验证结果 | ✅/⚠️ | 上下文完整性验证 |

## 检查清单

- [ ] AGENTS.md 是否存在且可读？
- [ ] 当前模块的 design.md 是否存在？
- [ ] 技术栈信息是否完整？
- [ ] 架构约束是否明确？
- [ ] 当前任务的目标和约束是否清晰？
- [ ] 依赖关系是否标注？
