# 设计文档生成

## 规范来源

> 本 Skill 基于 design.md 规范实现

## 核心原则

> ⚠️ **关键原则**：design.md 是代码变更的唯一真相来源

| 原则 | 说明 |
|------|------|
| **设计先行** | 开发前完成 design.md，设计领先于代码 |
| **单一真相** | design.md 是模块设计的唯一权威来源 |
| **持续同步** | 代码变更后同步更新 design.md |
| **边界明确** | 明确模块的输入、输出、依赖、被依赖 |

## 触发条件

- 需求提案通过后，需要生成模块级设计文档
- 用户要求"生成 design.md"或"写设计文档"
- 新模块开发前的设计阶段
- 设计评审前需要标准化文档

## 工作流

### Phase 1: 上下文收集

收集设计所需的上下文信息：

| 来源 | 内容 | 必要性 |
|------|------|--------|
| PRD 文档 | 功能范围、验收标准 | 必须 |
| AGENTS.md | 技术栈、架构约束、项目结构 | 必须 |
| 现有模块 design.md | 参考已有设计模式 | 推荐 |
| 上下文注入 | `/context-injector` 产出 | 推荐 |

### Phase 2: 模块边界定义

明确模块的边界：

```markdown
### 边界定义

- **输入**：[数据来源，如"前端请求"、"消息队列"]
- **输出**：[数据去向，如"前端响应"、"数据库持久化"]
- **依赖**：[依赖的其他模块]
- **被依赖**：[被哪些模块依赖]
```

### Phase 3: 按模板生成 design.md

按  模板结构生成：

```markdown
# [模块名] 设计文档

> 版本：v1.0
> 作者：[姓名]
> 更新日期：[日期]

## 1. 模块概述

### 1.1 职责定义
[模块负责的业务功能]

### 1.2 边界定义
- **输入**：[数据来源]
- **输出**：[数据去向]
- **依赖**：[依赖模块]
- **被依赖**：[被哪些模块依赖]

## 2. 技术设计

### 2.1 类图/架构图
[使用 Mermaid 绘制]

### 2.2 接口定义
[每个接口的请求/响应定义]

### 2.3 数据结构
[数据库表设计、核心数据对象]

### 2.4 核心逻辑
[核心业务逻辑说明]

## 3. 异常处理
[异常场景和处理方式]

## 4. 安全性
[安全措施]

## 5. 性能考虑
[性能优化点]

## 6. 扩展计划
[未来可能的扩展]

## 7. 变更记录
| 版本 | 日期 | 变更内容 | 变更人 |
```

### Phase 4: 架构约束校验

生成后校验设计是否符合架构约束：

- [ ] 依赖方向是否符合 AGENTS.md 中的架构约束？
- [ ] 是否存在循环依赖？
- [ ] 分层是否正确（Controller → Service → Repository）？
- [ ] 是否引入了未在 AGENTS.md 中声明的新依赖？

### Phase 5: 评审建议

输出设计评审建议：

```markdown
## 设计评审建议

### ✅ 设计亮点
- [亮点]

### ⚠️ 需要确认
- [需要架构师/技术负责人确认的点]

### 📋 评审前准备
- [ ] 确认接口定义与前端对齐
- [ ] 确认数据结构与 DBA 对齐
- [ ] 确认依赖关系无冲突
```

## 输入

| 输入项 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| PRD 文档 | 文本 | 是 | 10 要素 PRD |
| AGENTS.md | 文件 | 是 | 项目架构约束 |
| 模块名称 | 文本 | 是 | 要设计的模块名 |
| 参考模块 | 文本 | 否 | 参考已有模块的 design.md |

## 输出

| 输出项 | 格式 | 说明 |
|--------|------|------|
| design.md | Markdown | 模块级设计文档 |
| 架构约束校验 | ✅/❌ | 是否符合架构约束 |
| 评审建议 | Markdown | 设计评审前的准备建议 |

## 检查清单

- [ ] 模块职责是否单一明确？
- [ ] 边界定义是否清晰（输入/输出/依赖）？
- [ ] 接口定义是否完整（请求/响应/错误码）？
- [ ] 数据结构是否与现有表一致？
- [ ] 核心逻辑是否覆盖主流程和异常流程？
- [ ] 架构约束是否满足？
