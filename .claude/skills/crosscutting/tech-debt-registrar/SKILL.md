# 技术债务登记器

## 规范来源

> 本 Skill 基于 技术债务管理实现

## 触发条件

- code-review-expert 审查发现 P2 问题
- vibe-verify 验收发现 P2 问题
- fix-priority-triage 分诊后需要录入 P2 问题
- 月度技术债务盘点
- 超期债务告警
- 用户主动登记技术债务

## 工作流

### Phase 1: 债务录入

将 P2 问题标准化录入技术债务登记册：

```
债务录入
    │
    ├── 来源：code-review-expert P2 问题
    ├── 来源：vibe-verify P2 问题
    ├── 来源：fix-priority-triage P2 问题
    ├── 来源：entropy-manager 健康度告警
    └── 来源：用户手动录入

标准化字段：
    ├── ID：TD-NNN（自动递增）
    ├── 描述：问题简述
    ├── 位置：文件路径/模块名
    ├── 复杂度：低/中/高
    ├── 建议迭代：目标修复迭代
    ├── 状态：待处理/进行中/已修复/已关闭
    ├── 登记日期：YYYY-MM-DD
    └── 来源：审查/验收/分诊/手动
```

### Phase 2: 技术债务登记册

维护标准化的技术债务登记册：

```markdown
## 技术债务登记册

| ID | 描述 | 位置 | 复杂度 | 建议迭代 | 状态 | 登记日期 |
|----|------|------|--------|----------|------|----------|
| TD-001 | UserService 方法过长 | service/UserService.java | 中 | v2.1 | 待处理 | 2026-04-15 |
| TD-002 | 缺少订单模块单元测试 | module/order | 高 | v2.0 | 进行中 | 2026-04-10 |
| TD-003 | 魔法数字未提取常量 | util/DateUtils.java | 低 | v2.2 | 待处理 | 2026-04-18 |
```

### Phase 3: 债务优先级排序

按多维度对技术债务进行优先级排序：

```
优先级排序维度
    │
    ├── 影响范围
    │   ├── 全局影响（架构级）→ 最高优先
    │   ├── 模块影响（模块级）→ 高优先
    │   └── 局部影响（方法级）→ 低优先
    │
    ├── 修复成本
    │   ├── 低成本（<1天）→ 优先修复
    │   ├── 中成本（1-3天）→ 计划修复
    │   └── 高成本（>3天）→ 专项迭代
    │
    ├── 风险等级
    │   ├── 高风险（可能导致故障）→ 最高优先
    │   ├── 中风险（影响可维护性）→ 高优先
    │   └── 低风险（规范问题）→ 低优先
    │
    └── 超期程度
        ├── 超期 >2 个迭代 → 🚫 必须处理
        ├── 超期 1 个迭代 → ⚠️ 优先处理
        └── 未超期 → 按计划处理
```

### Phase 4: 超期债务跟踪

监控和告警超期未处理的技术债务：

```
超期检查
    │
    ├── 当前迭代 vs 建议迭代
    │   ├── 已超期 → 标记超期天数
    │   └── 未超期 → 正常跟踪
    │
    ├── 超期告警
    │   ├── 超期 1 个迭代 → ⚠️ 提醒
    │   ├── 超期 2 个迭代 → 🚫 必须处理
    │   └── 超期 3+ 个迭代 → 🚫 升级处理（纳入迭代强制项）
    │
    └── 处理建议
        ├── 低成本超期债务 → 立即修复
        ├── 高成本超期债务 → 纳入下次迭代
        └── 无需修复的债务 → 关闭并记录原因
```

### Phase 5: 债务生命周期管理

管理技术债务的完整生命周期：

```
债务生命周期
    │
    ├── 📝 待处理 → 新录入的债务
    │   └── 触发：P2 问题录入
    │
    ├── 🔧 进行中 → 已纳入迭代计划
    │   └── 触发：开发认领并开始修复
    │
    ├── ✅ 已修复 → 修复完成待验证
    │   └── 触发：代码提交并合并
    │
    └── 📋 已关闭 → 验证通过或不再需要
        ├── 触发：验证通过
        └── 触发：功能下线/需求变更
```

## 输入

| 输入项 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| 操作类型 | 枚举 | 否 | register（录入）/ list（查询）/ update（更新）/ report（报告），默认 list |
| 债务信息 | 对象 | 否 | 录入时需要：描述、位置、复杂度、建议迭代 |
| 债务 ID | 文本 | 否 | 更新/查询时需要 |

## 输出

| 输出项 | 格式 | 说明 |
|--------|------|------|
| 债务登记册 | 表格 | 完整的技术债务清单 |
| 优先级排序 | 表格 | 按多维度排序的债务列表 |
| 超期告警 | 表格 | 超期债务清单和处理建议 |
| 债务趋势 | 表格 | 新增/修复/净增趋势 |
| 修复建议 | 列表 | 本迭代建议修复的债务 |

## 检查清单

- [ ] P2 问题是否已录入技术债务登记册？
- [ ] 债务描述是否清晰可操作？
- [ ] 复杂度评估是否合理？
- [ ] 建议迭代是否明确？
- [ ] 是否有超期未处理的债务？
- [ ] 超期债务是否有处理计划？
- [ ] 已修复债务是否已验证关闭？
- [ ] 债务总量趋势是否可控？
