# 绝不执行规则（Never Rules）

> 以下操作绝对禁止，无论任何情况

## Git 操作

| 禁止操作 | 原因 | 替代方案 |
|----------|------|----------|
| 直接提交到 main 或生产分支 | 生产分支必须通过 MR + 代码审查 | 创建 feature 分支 → MR → 审查后合并 |
| 绕过代码审查直接合并 | 代码审查是质量门禁 | 走标准 MR 流程，至少 1 人审查 |

## 代码操作

| 禁止操作 | 原因 | 替代方案 |
|----------|------|----------|
| 删除整个模块或核心功能 | 不可逆的破坏性操作 | 先标记为 deprecated，经评审后再删除 |
| 暴露敏感信息（密钥、密码、Token） | 安全红线 | 使用环境变量或密钥管理服务 |

## 数据库操作

| 禁止操作 | 原因 | 替代方案 |
|----------|------|----------|
| 执行破坏性数据库操作 | 数据丢失风险 | 使用软删除（`isDelete = "Y"`），避免物理删除 |
| `DROP TABLE` | 不可逆 | 先备份数据，经 DBA 审批 |
| `DELETE` 无 `WHERE` 条件 | 全表删除风险 | 始终添加 WHERE 条件 + LIMIT |

## 安全操作

| 禁止操作 | 原因 | 替代方案 |
|----------|------|----------|
| 绕过安全检查机制 | 安全底线 | 遵循安全编码规范，修复安全问题而非绕过 |
| 硬编码密钥/密码 | 信息泄露风险 | 使用配置中心或环境变量 |
| SQL 拼接 | SQL 注入风险 | 使用参数化查询或 ORM |

## 处理方式

当用户要求执行 Never 级别操作时：

1. **拒绝** — 明确告知此操作被禁止
2. **说明** — 引用具体规则条目
3. **替代** — 提供合规的替代方案
4. **升级** — 如用户坚持，建议联系技术负责人或架构师审批
