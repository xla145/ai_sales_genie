# 源码借鉴优化专家

## 规范来源

> 本 Skill 参考 `workflow/verify/code-review-expert` 的阶段化工作流、输入输出约束、检查清单和门禁判定方式实现。

## 核心定位

当用户给出一个**参考源码**，并指定一个**目标项目**时，使用本 Skill。

本 Skill 的目标不是复制外部源码，而是提炼参考源码中可迁移的设计思想，并结合用户指定目标项目的业务目标、技术栈、现有结构和修改边界，形成可落地的优化方案。

**重要约束**：目标项目必须由用户明确指定。除非用户明确说“优化当前项目/当前仓库”，否则不得默认使用当前工作目录。

---

## 关键定义

| 概念 | 说明 | 是否可修改 |
|------|------|------------|
| 参考源码 | 用于学习设计思路、架构模式、流程和工程实践的源码、仓库、目录或片段 | 不可修改 |
| 目标项目 | 用户明确指定、需要被分析和优化的项目、目录、仓库或模块 | 经确认后可修改 |
| 优化范围 | 用户希望优化的具体方向，如前端、后端、Agent 流程、原型生成、数据库、工作流、整体架构 | 需确认 |

---

## 触发条件

- 用户提供参考源码地址，并要求学习后优化另一个项目
- 用户要求“参考这个项目的架构，改造指定项目”
- 用户要求“借鉴这个仓库的实现方式，优化目标项目的前端 / 后端 / 工作流”
- 用户要求学习某个项目的 agent、workflow、prompt、UI、API 或工程组织方式
- 用户提供本地源码目录，希望对指定目标项目进行对标优化

## 不触发条件

- 用户只想做普通代码审查：使用 `code-review-expert`
- 用户只想生成 FastAPI 后端代码：使用 `fastapi-backend-codegen`
- 用户只想生成 Vue 页面：使用 `vue-frontend-design`
- 用户只想整理需求或生成原型：使用对应需求/原型 Skill
- 用户没有提供参考源码，也没有指定目标项目：先澄清输入，不执行分析

## 工作流

### Phase 1: Preflight 输入与边界确认

必须先确认两类输入：

| 输入 | 必填 | 示例 | 处理规则 |
|------|------|------|----------|
| 参考源码 | 是 | GitHub URL、本地目录、代码片段、压缩包 | 只读分析，不修改 |
| 目标项目 | 是 | 当前仓库、本地路径、远程仓库、指定模块 | 用户明确授权后才可修改 |
| 优化范围 | 建议 | 前端、后端、Agent、工作流、原型生成、整体架构 | 不明确时先询问 |
| 执行模式 | 建议 | 仅分析 / 方案确认后实施 / 允许直接实施小改 | 不明确时默认先给方案 |

**边界情况：**

- **缺少参考源码**：要求用户补充参考源码地址、目录或片段。
- **缺少目标项目**：要求用户明确指定目标项目，不得默认当前工作目录。
- **目标项目写成“这个项目”但不明确**：询问是否指当前仓库或其它路径。
- **远程参考仓库较大**：优先只读查看 README、目录结构、关键文件；如需 clone，先征得用户同意。
- **许可证不明确**：提醒只能学习思想，不能复制源码实现。

---

### Phase 2: 参考源码只读研究

根据参考源码来源选择方式：

| 来源 | 处理方式 |
|------|----------|
| 远程仓库 URL | 优先使用只读方式查看 README、目录结构、配置、入口和核心模块 |
| 本地目录 | 只读查看目录结构、README、配置、入口、核心模块 |
| 代码片段 | 分析架构意图、抽象边界、流程组织和实现取舍 |
| 压缩包/文件 | 先确认是否允许读取，再分析关键文件 |

研究重点：

1. **核心目标**：参考项目主要解决什么问题。
2. **关键流程**：业务流、数据流、Agent 流、页面流或任务流如何组织。
3. **架构模式**：分层、模块化、状态管理、任务编排、插件化、事件驱动等。
4. **抽象边界**：哪些职责被拆开，哪些模块保持简单。
5. **工程实践**：配置、校验、测试、错误处理、日志、构建、发布等。
6. **不可迁移点**：技术栈不匹配、复杂度过高、业务不适配、许可证风险等。

**禁止事项：**

- 禁止修改参考源码。
- 禁止直接复制参考源码实现。
- 禁止为了“像参考项目”而忽略目标项目现实约束。

---

### Phase 3: 目标项目现状扫描

只扫描用户指定的目标项目。除非用户明确指定，否则不得把当前工作目录视为目标项目。

扫描内容：

| 方向 | 关注点 |
|------|--------|
| 前端 | views、components、stores、services、types、utils、router、样式体系 |
| 后端 | api、services、repositories、models、schemas、storage、任务队列 |
| Agent / Workflow | runner、skills、OpenSpec、prompt、任务编排、artifact 管理 |
| 数据库 | schema、migration、db model、repository、查询边界 |
| 工程化 | 配置、依赖、脚本、测试、lint、CI、目录组织 |

本阶段需要识别：

- 目标项目技术栈和主要分层
- 当前实现方式和关键痛点
- 与参考源码思想之间的差距
- 可最小化落地的改进点
- 可能影响的文件、模块和风险

---

### Phase 4: 可迁移思想分级

对参考源码中的设计思想进行分级：

| 等级 | 含义 | 处理方式 |
|------|------|----------|
| T0 - 强可迁移 | 与目标项目技术栈、业务目标和现有结构高度匹配 | 优先纳入方案 |
| T1 - 条件迁移 | 思路有价值，但需要裁剪或适配 | 给出适配方案和风险 |
| T2 - 不建议迁移 | 复杂度、成本、许可证或业务不匹配 | 明确说明不采用原因 |

必须输出“采用 / 裁剪采用 / 不采用”的判断，禁止泛泛而谈。

---

### Phase 5: 优化映射与门禁判定

在实施前输出映射表：

| 参考源码思想 | 迁移等级 | 目标项目现状 | 建议优化 | 预期收益 | 风险/代价 |
|-------------|----------|--------------|----------|----------|-----------|

门禁判定：

```
源码借鉴优化门禁
    │
    ├── 未指定目标项目 → 阻断，必须先确认目标项目
    │
    ├── 未授权修改目标项目 → 仅允许输出分析和方案
    │
    ├── 存在许可证/版权复制风险 → 禁止复制源码，仅保留思想级借鉴
    │
    ├── 多文件/架构级修改 → 必须先输出计划并等待用户确认
    │
    └── 单文件 Skill/文案/配置小改 → 可直接修改，但需说明改动
```

| 判定 | 条件 | 后续动作 |
|------|------|----------|
| BLOCKED | 缺少参考源码、缺少目标项目、权限不明确 | 先向用户澄清 |
| ANALYSIS_ONLY | 用户只要求分析或未授权修改 | 输出分析与方案，不改代码 |
| NEED_CONFIRMATION | 涉及多文件、架构、业务逻辑或风险较高修改 | 输出实施计划，等待确认 |
| APPLY_ALLOWED | 用户已授权，且改动范围明确 | 按计划实施并验证 |

---

### Phase 6: 实施计划

若需要修改目标项目，必须先给出实施计划：

```markdown
## 实施计划

**参考源码**：...
**目标项目**：...
**优化范围**：...
**执行模式**：...

### 修改范围
- ...

### 涉及文件
- ...

### 执行步骤
1. ...
2. ...
3. ...

### 验证方式
- ...

### 风险与回滚
- ...
```

未经用户确认，不得执行多文件架构修改。

---

### Phase 7: 实施优化

实施时遵守：

1. 优先编辑目标项目现有文件，不随意创建新框架或大目录。
2. 不引入未要求的新依赖。
3. 不做与优化目标无关的重构。
4. 不复制参考源码，只实现目标项目自己的版本。
5. 保持目标项目代码风格、命名和分层方式。
6. 保护用户已有未提交修改，遇到异常 git 状态先说明。
7. 若涉及前端 UI，按项目要求启动应用并实际验证。
8. 若涉及后端 API，运行相关测试或最小验证命令。
9. 若涉及 workflow / skill，检查触发条件、输入输出、边界情况和完成标准。

---

### Phase 8: 验证与最终报告

完成后输出简短报告：

```markdown
## 源码借鉴优化结果

**参考源码**：...
**目标项目**：...
**执行结果**：[已完成 / 仅分析 / 等待确认 / 阻断]

### 借鉴的设计思想
- ...

### 应用到目标项目的位置
- `path:line` ...

### 修改文件
- ...

### 验证方式
- ...

### 未实施项
- ...（原因）
```

## 输入

| 输入项 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| reference_source | URL / path / text | 是 | 参考源码、仓库、目录或代码片段 |
| target_project | path / repo / module | 是 | 用户指定的目标项目；不能默认当前目录 |
| optimization_scope | 枚举/文本 | 否 | frontend / backend / agent / workflow / prototype / architecture / full |
| mode | 枚举 | 否 | analysis_only / plan_then_apply / apply_allowed |
| constraints | 文本 | 否 | 不允许改动的范围、依赖限制、上线风险等 |

## 输出

| 输出项 | 格式 | 说明 |
|--------|------|------|
| 参考源码分析 | 要点列表 | 提炼参考源码的核心思路 |
| 可迁移思想分级 | T0/T1/T2 表格 | 区分采用、裁剪采用、不采用 |
| 目标项目映射 | 表格 | 将参考思想映射到目标项目现状 |
| 门禁判定 | BLOCKED/ANALYSIS_ONLY/NEED_CONFIRMATION/APPLY_ALLOWED | 决定是否可实施 |
| 实施计划 | Markdown | 修改范围、文件、步骤、验证、风险 |
| 最终报告 | Markdown | 修改内容、验证方式、未实施项 |

## 禁止事项

- 禁止在目标项目未明确时开始扫描或修改代码。
- 禁止默认使用当前工作目录作为目标项目。
- 禁止直接复制粘贴外部源码实现。
- 禁止绕过许可证或忽略版权风险。
- 禁止在未确认前进行多文件架构修改。
- 禁止为了“像参考项目”而破坏目标项目已有架构。
- 禁止把参考源码分析结果保存成长期项目文档，除非用户明确要求。
- 禁止把中间草稿暴露为最终交付物。

---

## 完成标准

本 Skill 执行完成时，应满足：

- 已明确参考源码和用户指定的目标项目。
- 已明确优化范围和执行模式。
- 已提炼参考源码的关键设计思想。
- 已区分 T0/T1/T2 迁移等级。
- 已将参考思想映射到目标项目具体模块。
- 若实施了修改，已完成必要验证。
- 最终回复清楚说明改了什么、为什么改、如何验证。
