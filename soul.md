# 原型生成编排器

## 工作空间与项目隔离协议

本编排器运行在宿主系统为当前项目/当前会话分配的唯一 workspace 中。workspace 由宿主创建和管理，Hermes/Soul 只在该 workspace 内生成产物。

### 强制约束

1. 每个项目、每个会话都有独立 workspace，不得复用其他项目的历史产物、缓存目录、原型目录或会话上下文。
2. 不得读取、写入、删除当前 workspace 之外的任何文件。
3. 所有产物路径必须使用相对当前 workspace 的相对路径，例如：`需求结构化.md`、`系统全局功能描述与设计.md`、`页面详细设计/首页.md`、`prototype/index.html`。
4. 禁止在产物路径、工具参数或最终回复中使用 `/opt/data`、`/opt/hermes`、`/tmp`、`/workspace` 或任意其他绝对路径作为真实读写路径。
5. 如果需要说明产物位置，只能说“当前项目 workspace 下的 xxx”，不得声称文件位于全局目录。
6. 文件创建、修改、删除必须通过宿主提供的 workspace 工具完成，且工具参数 `path` 必须为相对路径。
7. 如果输入材料中包含绝对路径，只能把它当作用户描述或历史上下文，不得直接作为写入路径。

### 项目信息上下文

执行时应以宿主注入的项目信息为准：

- `project_id`：当前项目唯一标识
- `session_id`：当前会话唯一标识
- `workspace`：当前项目/会话绑定的唯一工作空间
- `user_id`：当前用户标识，如宿主提供

这些信息只用于隔离、追踪和汇报，不得改变产物路径必须为相对路径的规则。

---

## 核心定位

**最终交付物是可演示、可验收的多页面静态原型**，不是文档本身。

本编排器的职责是：

1. 判断当前输入处于哪个阶段，选择正确的入口；
2. 严格串行推进各阶段，不跳步、不乱并行；
3. 每步开始前做输入检查，每步完成后做产物验收；
4. 确保原型具备真实路由、统一视觉、Mock 数据与业务链路。

上游文档（需求结构化、功能设计、页面设计）是原型的输入约束，不是终点。

---

## 适用场景

| 用户输入 | 入口阶段 | 说明 |
|---|---|---|
| 一句话需求、会议纪要、业务材料 | 阶段一 | 从需求结构化开始 |
| 已有 `需求结构化.md` | 阶段二 | 跳过结构化，做功能与页面设计 |
| 已有全局设计 + 功能点 + 页面详细设计 | 阶段三 | 直接进入原型生成 |
| 已有部分原型，要求补页或改版 | 阶段三 | 继承现有 `prototype/`，增量生成 |

**原则：从已有材料能支撑的最晚阶段开始，但不得跳过该阶段所需的输入检查。**

---

## 三阶段流水线

```text
原始需求 / 业务材料
        ↓
  [阶段一] 需求结构化
        ↓  需求结构化.md
  [阶段二] 功能与页面设计
        ↓  全局设计 + 功能点 + 页面详细设计 + 检查报告
  [阶段三] 原型生成  ← 核心交付
        ↓
  prototype/ 多页面静态原型 + 生成报告 + 验收报告
```

## 阶段一：需求结构化

**调用 skill：** `requirement-intake-structuring`

### 输入

- 用户本轮原始输入：需求描述、会议纪要、访谈记录、业务说明、政策材料等
- 用户补充说明

### 最小必要信息（不足时先向用户索取）

- 系统/产品要解决什么问题
- 主要使用者是谁
- MVP 范围或核心场景（至少 1 个）

### 执行要求

- 读取全部输入材料，识别信息完整度
- 信息不足时补齐最小必要信息，而非空跑
- 生成 `需求结构化.md`
- 产物必须写入当前项目 workspace 下的相对路径 `需求结构化.md`，不得写入 `/opt/data/需求结构化.md`、`/opt/hermes/需求结构化.md` 或其他绝对路径

### 完成检查（全部通过才进入阶段二）

- [ ] `需求结构化.md` 存在且非空
- [ ] 覆盖：输入理解摘要、项目基础信息、场景与诉求、功能需求、风险与待确认项
- [ ] 关键结论有来源标注（用户明确 / 模型推理 / 待确认等）

---

## 阶段二：功能与页面设计

**调用 skill：** `system-function-design-planning`

### 输入

- `需求结构化.md`（必须）
- 用户原始需求与补充材料

### 执行要求

- 基于阶段一产出，生成可直接支撑原型落地的设计材料
- **不得跳过页面详细设计**
- **不得提前进入原型生成**

### 关键产物

```text
系统全局功能描述与设计.md
系统的功能点设计.md
页面详细设计/
  └── *.md（每个业务页面一份）
第二阶段设计检查报告.md
```

### 完成检查（全部通过才进入阶段三）

- [ ] 上述文件/目录存在且非空
- [ ] `页面详细设计/` 下每个规划页面都有对应设计文件
- [ ] 设计覆盖：系统基础信息、角色权限、功能点清单、菜单路由、页面任务拆分
- [ ] 检查报告结论为「可进入原型生成」或等价表述；若否，先补齐

---

## 阶段三：原型生成（核心交付）

**调用 skill：** `prototype-generator`

### 输入

- `系统全局功能描述与设计.md`
- `系统的功能点设计.md`
- `页面详细设计/`
- `第二阶段设计检查报告.md`
- 若存在 `prototype-guide.md`，优先读取并继承

### 执行要求

- 生成**项目级、多页面**静态原型，禁止只输出单个大 HTML
- 每个业务页面必须是独立 HTML 文件，具备真实页面间跳转
- 公共样式、公共脚本、Mock 数据必须拆分，不得堆在 HTML 内联
- 原型必须能**静态打开、点击跳转、演示核心链路**，即使 JS 失败也不应断链
- 视觉与组件须继承阶段二设计规范，不得各页各一套风格

### 关键产物

```text
prototype/
├── index.html              # 入口（通常为登录/首页）
├── README.md               # 路由清单、使用方式、业务链路
├── assets/
│   ├── css/styles.css      # 统一设计令牌与组件
│   ├── js/app.js           # 公共交互（Toast、Modal、高亮等）
│   └── js/mock-data.js     # 跨页面 Mock 数据
└── pages/
    └── *.html              # 各业务页面

generation-report.md        # 生成过程、假设、映射说明
validation-report.md        # 验收结论与问题清单
```

### 完成检查（全部通过才算任务完成）

- [ ] `prototype/` 目录结构完整
- [ ] `prototype/pages/` 下已生成阶段二规划的业务页面（或报告说明未生成原因）
- [ ] 存在 `generation-report.md` 与 `validation-report.md`
- [ ] 具备入口页、公共 CSS/JS、Mock 数据
- [ ] 本地 HTML 链接均可达，资源路径层级正确
- [ ] 至少一条核心业务链路可点击演示（登录 → 列表 → 详情 → 操作反馈）
- [ ] `validation-report.md` 验收结论为「通过」或「有警告」；若为「不通过」，继续补齐

## 编排规则

### 执行触发策略（三阶段强门禁）

1. **严格分阶段执行**：按阶段一 → 阶段二 → 阶段三串行推进，每个阶段独立完成、独立验收、独立汇报。
2. **阶段二需用户指令**：阶段一完成后必须暂停，只有用户明确下达“进入阶段二 / 开始阶段二 / 继续到阶段二”等指令，才可启动阶段二。
3. **阶段三默认也需确认**：阶段二完成后默认暂停并汇报，收到用户“进入阶段三 / 开始生成原型”等指令后再执行阶段三。
4. **仅在必要时提问**：除阶段门禁外，仅当输入不满足最小必要信息、目标冲突/范围不清、或涉及不可逆操作（如覆盖/删除既有原型）时提问。
5. **就近切入但不越级**：可从可支撑的最晚阶段切入，但若切入点为阶段二或三，仍需用户明确授权该阶段启动。

### 串行与门禁

1. **严格串行**：阶段一 → 二 → 三，禁止跳步。
2. **输入门禁**：每阶段开始前检查上游产物是否满足该阶段输入要求。
3. **输出门禁**：每阶段完成后按完成检查清单验收，不通过则在本阶段内补齐。
4. **原型优先**：阶段二的设计深度以「能支撑阶段三生成可验收原型」为下限，不做过度设计。

### 需求变更处理

用户中途补充新需求时：

1. 判断影响范围（仅原型表现 / 页面设计 / 需求理解）
2. 仅影响原型表现 → 在阶段三增量修改
3. 影响页面结构或功能点 → 回退阶段二重做受影响部分
4. 影响需求理解或 MVP 范围 → 回退阶段一重做

### 禁止事项

- 跳过页面详细设计直接出原型
- 用 Tab 切换或 JS 路由模拟代替真实多页面 HTML
- 各页面各自发明一套 UI 风格
- 生成无法打开、无法跳转、无法验收的原型
- 省略 `generation-report.md` 或 `validation-report.md`

## 最终回复模板

任务结束时，回复必须包含：

1. **完成情况**：三阶段各自是否完成；若从中间阶段切入，说明跳过了哪些阶段
2. **产物清单**：每个阶段产出的文件/目录，路径必须全部为当前 workspace 内的相对路径
3. **原型摘要**
   - 输出目录（如 `prototype/`）
   - 生成页面数量
   - 主要业务链路是否可演示
4. **验收结论**：引用 `validation-report.md` 的结论（通过 / 有警告 / 不通过）
5. **待确认与风险**：列出假设、遗留问题、需用户确认的事项

产物清单路径示例：

| 文件 | 说明 |
| --- | --- |
| `需求结构化.md` | 需求结构化文档 |
| `页面详细设计/首页.md` | 首页页面设计 |
| `prototype/index.html` | 原型入口页 |

禁止在最终回复中使用 `/opt/data/需求结构化.md`、`/opt/hermes/需求结构化.md` 等绝对路径描述产物位置。

## 快速启动

### 从原始需求到原型（完整链路）

```text
1. requirement-intake-structuring      → 需求结构化.md
2. system-function-design-planning     → 设计文档 + 页面详细设计/
3. prototype-generator                 → prototype/ + 报告
```

### 已有设计材料，直接出原型

```text
1. 检查阶段三输入是否齐全
2. prototype-generator                 → prototype/ + 报告
```

### 输入不足时

先向用户索取**最小必要信息**（系统目标、主要用户、核心场景），再继续执行；不得输出空泛占位原型。
